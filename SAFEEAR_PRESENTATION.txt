EXECUTIVE SUMMARY
================================================================================

SafeEar is a production-grade mobile threat detection application for Android
that uses advanced speaker embedding technology to identify unknown speakers
and alert parents/guardians of potential threats.

KEY VALUE PROPOSITION:
- Real-time acoustic threat detection using neural networks
- Two-device synchronized monitoring (Parent watches, Child records)
- Background operation with GPS location integration
- User-friendly Material 3 UI with Jetpack Compose
- Role-based navigation for parent and child scenarios

USE CASE:
Parents can monitor their child's environment by having the child's device
listen passively in the background, recording snippets of audio and analyzing
them for "stranger" voices. When an unknown speaker is detected with high
confidence, an alert is fired to the parent device with audio clip and location.

   - Dev token-based login (format: "dev:alias")
   - Role selection: Parent Device OR Child Device
   - Persistent storage via Android DataStore
   - Token refresh and lifecycle management

2. PARENT MONITORING DASHBOARD
   - Real-time alerts feed (poll every 15 seconds)
   - Alert acknowledgment with audio clip playback
   - Location display on alerts (latitude, longitude)
   - Speaker confidence scores
   - Historical alert review

3. SPEAKER ENROLLMENT & MANAGEMENT
   - Record trusted speakers for baseline embeddings
   - Upload speaker samples (WAV format)
   - List all enrolled speakers
   - Rename speakers for easy identification
   - Delete speakers from database

4. CHILD MODE MONITORING
   - One-tap START/STOP recording toggle
   - Continuous background audio capture (1.5s chunks @ 16kHz)
   - Auto-upload to backend for real-time analysis
   - GPS location tracking with each chunk
   - Live telemetry display (confidence, decision)
   - Foreground service to maintain recording while backgrounded

5. REAL-TIME NOTIFICATIONS & EVENTS
   - WebSocket connection for instant event delivery
   - Push notifications via Firebase Cloud Messaging (FCM)
   - WorkManager background refresh for alerts

6. SETTINGS & ACCOUNT MANAGEMENT
   - Device info (Device ID, Device Name, Role)
   - App version and build information
   - Permissions management (Microphone, Location, etc.)
   - Logout and account switching

BACKEND API INTEGRATION
================================================================================

BASE ENDPOINTS
──────────────

Authentication:
- POST /auth/google          → Login with id_token
- POST /auth/refresh         → Refresh access token
- POST /auth/logout          → Revoke tokens + cleanup

Device Registration (Critical for Demo):
- POST /devices              → Register device (parent_device or child_device)
  Input: { device_id, role, installation_id, device_name }
  Output: { id, role, installation_id, device_name, created_at }

Speaker Management (Parent Mode):
- POST /enroll/speaker       → Enroll new speaker (multipart: WAV file)
- GET /enroll/speakers       → List all enrolled speakers
- PATCH /enroll/speakers/{id} → Rename speaker
- DELETE /enroll/speakers/{id} → Delete speaker

Detection & Monitoring (Child Mode):
- POST /detect/chunk         → Upload audio chunk (child → parent backend)
  Input: { chunk_data (WAV), latitude, longitude, timestamp }
  Output: { decision, confidence, alert_fired, alert_id }
- POST /detect/location      → Update GPS location
- DELETE /detect/session     → End detection session on logout/role change

Alerts (Parent Mode):
- GET /alerts                → List alerts (paginated, filterable)
- POST /alerts/{id}/ack      → Acknowledge alert (mark as reviewed)
- GET /alerts/{id}/clip      → Download alert audio clip (wav file)

Real-Time Events (WebSocket):
- WS /ws/events              → Bi-directional event stream
  Output messages:
  - { type: "alert", data: { alert_id, decision, confidence, ... } }
  - { type: "detection", data: { decision, confidence, ... } }
  - { type: "speaker_recognized", data: { speaker_id, speaker_name, ... } }


REQUEST/RESPONSE AUTHENTICATION
───────────────────────────────

Token Flow:
1. POST /auth/google (dev:alias) → Receives { access_token, refresh_token, expires_in }
2. Access token stored securely in DataStore
3. All subsequent requests include: Authorization: Bearer {access_token}
4. If 401 received → AuthInterceptor triggers TokenAuthenticator
5. TokenAuthenticator calls POST /auth/refresh → New access_token
6. Original request retried with new token

Headers (Standard):
- Authorization: Bearer {access_token}
- Content-Type: application/json (or multipart/form-data for file uploads)
- User-Agent: SafeEar/{version} Android/{sdk_version}


ERROR HANDLING & RECOVERY
─────────────────────────

Network Errors:
- No connectivity → Show "Offline" indicator, queue requests
- Timeout (30s default) → Show "Connection timeout" error, allow retry
- DNS failure → Show "Cannot reach server", provide manual refresh

API Errors:
- 400 Bad Request → Display validation error to user
- 401 Unauthorized → Refresh token, auto-retry (once)
- 403 Forbidden → Show "Access denied" or logout on child mode
- 404 Not Found → Show "Resource not found"
- 500 Server Error → Show "Server error" with retry option
- 503 Service Unavailable → Show "Service temporarily unavailable"

User Feedback:
- Snackbar messages for errors
- Dialog for fatal errors requiring app restart
- Loading spinners during long operations
- Toast notifications for quick feedback

DATA FLOW EXAMPLES (FOR DEMO WALKTHROUGH)
================================================================================

SCENARIO 1: Parent Setting Up Monitoring
─────────────────────────────────────────

1. User launches app
   → AuthViewModel checks TokenManager for saved token
   → If token exists: direct to role-based screen
   → If no token: show LoginScreen

2. LoginScreen: User enters "dev:test_parent"
   → Clicks "Login" button
   → AuthViewModel calls AuthRepository.login("dev:test_parent")
   → ApiService.login() → POST /auth/google
   → Response: { access_token, refresh_token, expires_in }
   → TokenManager saves tokens to DataStore (encrypted)
   → Navigation → DeviceRegistrationScreen

3. DeviceRegistrationScreen: User selects "Parent Device"
   → Clicks "Register Device"
   → DeviceRegistrationViewModel calls DeviceRepository.registerDevice("parent_device")
   → ApiService.queryDevices() → POST /devices with role="parent_device"
   → Response: { device_id, installation_id, role, device_name }
   → TokenManager saves role="parent_device" to DataStore
   → AlertsViewModel initializes
   → Navigation → ParentDashboard

4. ParentDashboard loads with Alerts Tab (default)
   → AlertsViewModel calls AlertsRepository.getAlerts()
   → ApiService.getAlerts() → GET /alerts
   → Response: List<AlertDto>
   → StateFlow emits alerts list
   → AlertsScreen collects and renders alert cards
   
5. RealtimeSocketManager acquires connection
   → WebSocket opens to WS /ws/events
   → Subscribes to real-time alert notifications
   → When new alert arrives: WebSocket message → parsed → StateFlow update → UI refresh


SCENARIO 2: Child Device Recording & Uploading
───────────────────────────────────────────────

1. Child device launches app, logs in with "dev:test_child"
   → Same auth flow as parent
   → DeviceRegistrationScreen: select "Child Device"
   → Navigation → ChildMonitoringScreen

2. ChildMonitoringScreen starts lifecycle
   → ChildMonitoringViewModel init { }:
   ├─ DeviceRepository.startDevicePolling() — polling begins
   ├─ RealtimeSocketManager.acquire() — WebSocket connection
   └─ UI shows "SafeEar Idle"

3. User taps the MIC button → START
   → ChildMonitoringViewModel.toggleMonitoring(true)
   → Starts DetectionService (foreground service)
   → DetectionService starts MediaRecorder
   ├─ Captures 16 kHz PCM audio
   ├─ Records in 1.5-second chunks
   └─ Each chunk auto-uploaded

4. For each audio chunk (every 1.5 sec):
   → DetectionService collects 1.5s of audio
   → Encodes to WAV format
   → Gets current GPS location (FusedLocationProvider)
   → Calls DetectionRepository.uploadChunk(wavData, lat, lon)
   → ApiService.uploadChunk() → POST /detect/chunk
   → Response: { decision, confidence, alert_fired, alert_id }
   → ChildMonitoringViewModel updates currentDecision StateFlow
   → ChildMonitoringScreen re-renders metrics
   → If alert_fired: show alert notification + add to activityLogs

5. UI updates real-time:
   → Confidence score refreshes every 1.5 sec
   → Waveform bars animate (simulated or actual audio level)
   → Activity log appends new row
   → If offline: show "Connection lost" indicator

6. User taps MIC button → STOP
   → ChildMonitoringViewModel.toggleMonitoring(false)
   → Stops DetectionService
   → Stops MediaRecorder
   → Stops chunk upload loop
   → UI transitions to "SafeEar Idle"


SCENARIO 3: Parent Enrolls a Speaker
──────────────────────────────────────

1. ParentDashboard on Speakers Tab
   → SpeakerListScreen displays enrolled speakers
   → User taps "+  Enroll New Speaker" FAB

   → Option B: Upload existing file

      ├─ File picker opens

SCENARIO 4: Parent Reviews Alert & Plays Audio
────────────────────────────────────────────────

1. ParentDashboard on Alerts Tab
   → AlertsScreen displays list of alerts
   → Each alert card shows: timestamp, confidence, location

2. User taps an alert card
   → Alert expands or navigates to detail view
   → Details shown:
      ├─ Speaker status: "Unknown Speaker" or "John (familiar)"
      ├─ Timestamp: "Today at 2:15 PM"
      ├─ Confidence: 94% (green badge)
      ├─ Location: "Home" (address or coordinates)
      ├─ Decision: "stranger_candidate"
      └─ Action buttons: Play | Download | Acknowledge | Delete

3. User taps "Play" button
   → AlertsViewModel calls AlertsRepository.downloadClip(alertId)
   → ApiService.downloadClip() → GET /alerts/{id}/clip
   → Response: WAV audio file (binary)
   → Saved to temp cache
   → AudioPlayer (ExoPlayer) configured with URI
   → Playback starts
   → Progress slider shows playback position
   → User can scrub, pause, resume

4. User taps "Acknowledge" button
   → AlertsViewModel calls AlertsRepository.acknowledgeAlert(alertId)
   → ApiService.ackAlert() → POST /alerts/{id}/ack
   → Response: { id, ack_status, ack_timestamp }
   → Alert marked as "reviewed" in backend
   → Alert card visual changes (e.g., becomes grayed out)
   → Confirmation: "Alert acknowledged"

5. User taps "Delete" button
   → Confirmation dialog: "Delete this alert? This cannot be undone."
   → User confirms
   → AlertsViewModel calls AlertsRepository.deleteAlert(alertId)
   → ApiService.deleteAlert() → DELETE /alerts/{id}
   → Alert removed from list
   → Alert list UI updated automatically

RECENT FIXES & IMPROVEMENTS (April 2026)
================================================================================

STABILITY IMPROVEMENTS
──────────────────────

1. Polling Synchronization (Critical Fix)
   Problem: Duplicate /devices polling loop running simultaneously
   Impact: High CPU/battery drain, cache thrashing
   Solution: 
   - Added synchronized(pollingLock) guard in DeviceRepository
   - Single polling coroutine guaranteed per app instance
   - Prevents concurrent start/stop races

2. WebSocket Lifecycle Management (Critical Fix)
   Problem: WebSocket connection not established, delivered=0 always
   Impact: Parent receives ~0 real-time alerts
   Solution:
   - Added RealtimeSocketManager.acquire() in AlertsViewModel
   - Paired with release() in finally block
   - Ensures proper ref-count lifecycle

3. Role-Scoped Device Identity (Critical Fix)
   Problem: Parent/child role would flip on same device
   Impact: Child switches to "parent" unexpectedly, features disappear
   Solution:
   - Added INSTALLATION_ID_PARENT_KEY and INSTALLATION_ID_CHILD_KEY
   - Preserve user-selected role locally after /devices/upsert
   - Prevent backend response from overwriting role
